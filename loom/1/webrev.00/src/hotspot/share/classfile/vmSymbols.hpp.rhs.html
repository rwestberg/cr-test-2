 726 // Here are all the intrinsics known to the runtime and the CI.
 727 // Each intrinsic consists of a public enum name (like _hashCode),
 728 // followed by a specification of its klass, name, and signature:
 729 //    template(<id>,  <klass>,  <name>, <sig>, <FCODE>)
 730 //
 731 // If you add an intrinsic here, you must also define its name
 732 // and signature as members of the VM symbols.  The VM symbols for
 733 // the intrinsic name and signature may be defined above.
 734 //
 735 // Because the VM_SYMBOLS_DO macro makes reference to VM_INTRINSICS_DO,
 736 // you can also define an intrinsic's name and/or signature locally to the
 737 // intrinsic, if this makes sense.  (It often does make sense.)
 738 //
 739 // For example:
 740 //    do_intrinsic(_foo,  java_lang_Object,  foo_name, foo_signature, F_xx)
 741 //     do_name(     foo_name, "foo")
 742 //     do_signature(foo_signature, "()F")
 743 // klass      = vmSymbols::java_lang_Object()
 744 // name       = vmSymbols::foo_name()
 745 // signature  = vmSymbols::foo_signature()
 746 //
 747 // The name and/or signature might be a "well known" symbol
 748 // like "equal" or "()I", in which case there will be no local
 749 // re-definition of the symbol.
 750 //
 751 // The do_class, do_name, and do_signature calls are all used for the
 752 // same purpose:  Define yet another VM symbol.  They could all be merged
 753 // into a common 'do_symbol' call, but it seems useful to record our
 754 // intentions here about kinds of symbols (class vs. name vs. signature).
 755 //
 756 // The F_xx is one of the Flags enum; see below.
 757 //
 758 // for Emacs: (let ((c-backslash-column 120) (c-backslash-max-column 120)) (c-backslash-region (point) (point-max) nil t))
 759 //
 760 //
 761 // There are two types of intrinsic methods: (1) Library intrinsics and (2) bytecode intrinsics.
 762 //
 763 // (1) A library intrinsic method may be replaced with hand-crafted assembly code,
 764 // with hand-crafted compiler IR, or with a combination of the two. The semantics
 765 // of the replacement code may differ from the semantics of the replaced code.
 766 //
 767 // (2) Bytecode intrinsic methods are not replaced by special code, but they are
 768 // treated in some other special way by the compiler. For example, the compiler
 769 // may delay inlining for some String-related intrinsic methods (e.g., some methods
 770 // defined in the StringBuilder and StringBuffer classes, see
 771 // Compile::should_delay_string_inlining() for more details).
 772 //
 773 // Due to the difference between the semantics of an intrinsic method as defined
 774 // in the (Java) source code and the semantics of the method as defined
 775 // by the code in the VM, intrinsic methods must be explicitly marked.
 776 //
 777 // Intrinsic methods are marked by the jdk.internal.HotSpotIntrinsicCandidate
 778 // annotation. If CheckIntrinsics is enabled, the VM performs the following
 779 // checks when a class C is loaded: (1) all intrinsics defined by the VM for
 780 // class C are present in the loaded class file and are marked;
 781 // (2) an intrinsic is defined by the VM for all marked methods of class C;
 782 // (3) check for orphan methods in class C (i.e., methods for which the VM
 783 // declares an intrinsic but that are not declared for the loaded class C.
 784 // Check (3) is available only in debug builds.
 785 //
 786 // If a mismatch is detected for a method, the VM behaves differently depending
 787 // on the type of build. A fastdebug build exits and reports an error on a mismatch.
 788 // A product build will not replace an unmarked library intrinsic method with
 789 // hand-crafted code, that is, unmarked library intrinsics are treated as ordinary
 790 // methods in a product build. The special treatment of a bytecode intrinsic method
 791 // persists even if the method not marked.
 792 //

 794 // annotate the method in the source code. The list below contains all
 795 // library intrinsics followed by bytecode intrinsics. Please also make sure to
 796 // add the declaration of the intrinsic to the approriate section of the list.
